# CT 101 — Grafana + Prometheus

> Este contenedor centraliza el stack de monitorización del laboratorio. Prometheus actúa como motor de recolección de métricas, Node Exporter expone métricas del sistema operativo de cada CT, y Grafana proporciona la capa de visualización. Todo corre en un único LXC privilegiado dentro de la VLAN 30 (SOC).

## Prometheus

### Versión

> Prometheus es el motor de scraping y almacenamiento de métricas. Corre como binario gestionado por systemd (no en Docker) para mayor estabilidad y control directo sobre el proceso.

`2.45.0` — instalado como binario en `/opt/prometheus/`

> ⚠️ Versión desactualizada. La actual es 3.x. Planear actualización.

### Servicio systemd `/etc/systemd/system/prometheus.service`

> Define cómo systemd arranca y gestiona el proceso de Prometheus. El flag `--web.enable-lifecycle` es especialmente útil: permite recargar la configuración en caliente sin necesidad de reiniciar el servicio ni perder métricas.

```ini
[Unit]
Description=Prometheus
After=network.target

[Service]
Type=simple
ExecStart=/opt/prometheus/prometheus \
  --config.file=/opt/prometheus/prometheus.yml \
  --storage.tsdb.path=/opt/prometheus/data \
  --web.listen-address=0.0.0.0:9090 \
  --web.enable-lifecycle
Restart=always

[Install]
WantedBy=multi-user.target
```

> `--web.enable-lifecycle` permite recargar configuración sin reiniciar via `POST http://localhost:9090/-/reload`

### Configuración `/opt/prometheus/prometheus.yml`

> Archivo de configuración principal de Prometheus. Define cada `job` de scraping: qué endpoints monitorizar y con qué frecuencia. El intervalo global de 15s significa que Prometheus consulta cada target cada 15 segundos para obtener sus métricas actualizadas.

```yaml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

alerting:
  alertmanagers:
    - static_configs:
        - targets:

rule_files:

scrape_configs:
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  - job_name: 'node_exporter'
    static_configs:
      - targets:
        - '10.1.1.66:9100'    # CT101 Grafana-Prometheus
        - '10.1.1.34:9100'    # CT103 DNS
        - '10.1.1.35:9100'    # CT105 Nginx
        - '10.1.1.36:9100'    # CT106 Suricata
        - '10.1.1.37:9100'    # CT104 SOAR

  - job_name: 'bind_exporter'
    static_configs:
      - targets:
        - '10.1.1.34:9119'    # CT103 Bind9

  - job_name: 'suricata_node'
    static_configs:
      - targets:
        - '10.1.1.36:9100'    # CT106 Suricata
```

## Node Exporter

### Versión

> Node Exporter es el agente estándar de Prometheus para exponer métricas del sistema operativo: CPU, memoria, disco, red, temperatura, etc. Se instala en cada contenedor que se quiere monitorizar y escucha en el puerto `9100`. Prometheus lo consulta periódicamente para recoger esas métricas.

`v1.8.2` — instalado como binario en `/opt/node_exporter/`

### Servicio systemd `/etc/systemd/system/node_exporter.service`

> Configuración mínima del servicio. Node Exporter no requiere parámetros adicionales para el uso básico: arranca, expone las métricas del sistema en `:9100` y systemd se encarga de reiniciarlo si cae.

```ini
[Unit]
Description=Node Exporter
After=network.target

[Service]
Type=simple
ExecStart=/opt/node_exporter/node_exporter
Restart=always

[Install]
WantedBy=multi-user.target
```

> Node Exporter instalado en CT 101, 103, 104, 105 y 106. Escucha en puerto `9100`.

## Grafana

### Versión

> Grafana es la herramienta de visualización del stack. Corre en Docker (a diferencia de Prometheus y Node Exporter) para facilitar las actualizaciones y el aislamiento. Se conecta a Prometheus como datasource y permite construir dashboards con los datos recopilados.

`12.4.1` — corriendo en Docker

### Docker Compose `/root/grafana/docker-compose.yml`

> Define el contenedor Docker de Grafana. El volumen `grafana-storage` persiste los dashboards y configuraciones entre reinicios. La carpeta `./provisioning` permite preconfigurar datasources y dashboards automáticamente al arrancar, sin intervención manual.

```yaml
services:
  grafana:
    image: grafana/grafana-oss:12.4.1
    container_name: grafana
    restart: unless-stopped
    ports:
      - "3000:3000"
    volumes:
      - grafana-storage:/var/lib/grafana
      - ./provisioning:/etc/grafana/provisioning
    environment:
      - GF_SECURITY_ADMIN_USER=admin
      - GF_SECURITY_ADMIN_PASSWORD=changeme
      - GF_USERS_ALLOW_SIGN_UP=false

volumes:
  grafana-storage:
```

> ⚠️ Cambiar `GF_SECURITY_ADMIN_PASSWORD` en producción.

### Provisioning datasource `/root/grafana/provisioning/datasources/prometheus.yml`

> Este archivo configura automáticamente Prometheus como datasource al iniciar Grafana. Gracias al provisioning, no hace falta entrar en la UI para añadir el datasource manualmente: está preconfigurado y marcado como predeterminado.

```yaml
apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://10.1.1.66:9090
    isDefault: true
    editable: true
```

## Comandos útiles

> Referencia rápida de comandos para el día a día: verificar el estado de los servicios, recargar configuración sin downtime, reiniciar componentes y consultar los targets activos directamente desde la API de Prometheus.

```bash
# Estado de servicios
systemctl status prometheus
systemctl status node_exporter
docker ps

# Recargar config Prometheus sin reiniciar
curl -X POST http://localhost:9090/-/reload

# Reiniciar servicios
systemctl restart prometheus
systemctl restart node_exporter
cd /root/grafana && docker compose restart

# Ver logs
docker logs grafana
journalctl -u prometheus -f
journalctl -u node_exporter -f

# Verificar targets Prometheus
curl -s http://localhost:9090/api/v1/targets | python3 -c "
import json,sys
data=json.load(sys.stdin)
for t in data['data']['activeTargets']:
    print(t['labels']['job'], '-', t['labels']['instance'], '-', t['health'])
"
```
