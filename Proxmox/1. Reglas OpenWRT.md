## Interfaces de red

El CT dispone de dos interfaces físicas (virtuales a nivel Proxmox):

- **`eth0`** conectada al bridge `vmbr0` → actúa como **WAN** (acceso a internet/red física).
- **`eth1`** conectada al bridge `vmbr1` → actúa como **trunk de VLANs** hacia la red interna del laboratorio.

Las VLANs se crean como subinterfaces de `eth1` con etiquetado 802.1Q. Cada VLAN tiene su propio segmento `/27` (30 hosts útiles por segmento), lo que permite aislar los distintos entornos del SOC.

| Interfaz | Bridge Proxmox | Rol | IP |
|----------|---------------|-----|----|
| `eth0` | `vmbr0` | WAN (red externa) | `192.168.3.200/24` (estática) |
| `eth1` | `vmbr1` | LAN trunk (VLANs SOC) | — |
| `eth1.10` | `vmbr1` tag 10 | VLAN 10 — DMZ | `10.1.1.1/27` |
| `eth1.20` | `vmbr1` tag 20 | VLAN 20 — Servicios | `10.1.1.33/27` |
| `eth1.30` | `vmbr1` tag 30 | VLAN 30 — SOC | `10.1.1.65/27` |
| `eth1.40` | `vmbr1` tag 40 | VLAN 40 — Producción | `10.1.1.97/27` |
| `eth1.50` | `vmbr1` tag 50 | VLAN 50 — ??? | `10.1.1.129/27` |

### Esquema topológico

```
Internet
   ↓
Router (192.168.3.1)
   ↓
Proxmox vmbr0 (192.168.3.200)
   ↓
OpenWRT CT100
   ├── eth0  → 192.168.3.200/24  (WAN — estática)
   └── eth1  → trunk VLANs
         ├── eth1.10 → 10.1.1.1/27    (VLAN 10 DMZ)
         ├── eth1.20 → 10.1.1.33/27   (VLAN 20 Servicios)
         ├── eth1.30 → 10.1.1.65/27   (VLAN 30 SOC)
         ├── eth1.40 → 10.1.1.97/27   (VLAN 40 Producción)
         └── eth1.50 → 10.1.1.129/27  (VLAN 50 ???)
```

## ⚙️ Configuración de red (UCI)

OpenWRT gestiona la configuración de red a través de **UCI** (*Unified Configuration Interface*). Los siguientes comandos definen las interfaces WAN y las cinco VLANs internas con IPs estáticas. Una vez aplicados, se persisten con `uci commit` y se activan reiniciando el servicio de red.

>  Los cambios en `uci` no son efectivos hasta ejecutar `uci commit network` seguido de `service network restart`.

```bash
# WAN — IP estática
uci set network.wan.device='eth0'
uci set network.wan.proto='static'
uci set network.wan.ipaddr='192.168.3.200'
uci set network.wan.netmask='255.255.255.0'
uci set network.wan.gateway='192.168.3.1'

# VLAN 10 — DMZ
uci set network.vlan10=interface
uci set network.vlan10.device='eth1.10'
uci set network.vlan10.proto='static'
uci set network.vlan10.ipaddr='10.1.1.1'
uci set network.vlan10.netmask='255.255.255.224'

# VLAN 20 — Servicios
uci set network.vlan20=interface
uci set network.vlan20.device='eth1.20'
uci set network.vlan20.proto='static'
uci set network.vlan20.ipaddr='10.1.1.33'
uci set network.vlan20.netmask='255.255.255.224'

# VLAN 30 — SOC
uci set network.vlan30=interface
uci set network.vlan30.device='eth1.30'
uci set network.vlan30.proto='static'
uci set network.vlan30.ipaddr='10.1.1.65'
uci set network.vlan30.netmask='255.255.255.224'

# VLAN 40 — Producción
uci set network.vlan40=interface
uci set network.vlan40.device='eth1.40'
uci set network.vlan40.proto='static'
uci set network.vlan40.ipaddr='10.1.1.97'
uci set network.vlan40.netmask='255.255.255.224'

# VLAN 50 — ???
uci set network.vlan50=interface
uci set network.vlan50.device='eth1.50'
uci set network.vlan50.proto='static'
uci set network.vlan50.ipaddr='10.1.1.129'
uci set network.vlan50.netmask='255.255.255.224'

uci commit network
service network restart
```

## DHCP

### Configuración por VLAN

El servidor DHCP lo gestiona **dnsmasq**, incluido por defecto en OpenWRT. Se configura un pool independiente por cada VLAN, todos con un `start=10` para evitar colisiones con las IPs estáticas asignadas manualmente a los contenedores (que ocupan las primeras posiciones de cada subred).

Con `limit=20` se permiten hasta 20 clientes DHCP por VLAN, lo cual encaja con el tamaño `/27` de cada segmento (30 hosts útiles).

| VLAN | Interfaz | Rango DHCP | Lease |
|------|----------|------------|-------|
| VLAN 10 | `vlan10` | `10.1.1.10` — `10.1.1.30` | 12h |
| VLAN 20 | `vlan20` | `10.1.1.43` — `10.1.1.63` | 12h |
| VLAN 30 | `vlan30` | `10.1.1.75` — `10.1.1.95` | 12h |
| VLAN 40 | `vlan40` | `10.1.1.107` — `10.1.1.127` | 12h |
| VLAN 50 | `vlan50` | `10.1.1.139` — `10.1.1.159` | 12h |

> El `start=10` evita colisiones con IPs estáticas asignadas a los contenedores. El rango dinámico comienza en el décimo host de cada subred.

### Comandos aplicados

```bash
uci set dhcp.vlan10=dhcp
uci set dhcp.vlan10.interface='vlan10'
uci set dhcp.vlan10.start='10'
uci set dhcp.vlan10.limit='20'
uci set dhcp.vlan10.leasetime='12h'

uci set dhcp.vlan20=dhcp
uci set dhcp.vlan20.interface='vlan20'
uci set dhcp.vlan20.start='10'
uci set dhcp.vlan20.limit='20'
uci set dhcp.vlan20.leasetime='12h'

uci set dhcp.vlan30=dhcp
uci set dhcp.vlan30.interface='vlan30'
uci set dhcp.vlan30.start='10'
uci set dhcp.vlan30.limit='20'
uci set dhcp.vlan30.leasetime='12h'

uci set dhcp.vlan40=dhcp
uci set dhcp.vlan40.interface='vlan40'
uci set dhcp.vlan40.start='10'
uci set dhcp.vlan40.limit='20'
uci set dhcp.vlan40.leasetime='12h'

uci set dhcp.vlan50=dhcp
uci set dhcp.vlan50.interface='vlan50'
uci set dhcp.vlan50.start='10'
uci set dhcp.vlan50.limit='20'
uci set dhcp.vlan50.leasetime='12h'

uci commit dhcp
service dnsmasq restart
```

### DNS relay

En lugar de resolver DNS localmente, dnsmasq está configurado como **relay** hacia el servidor **Bind9** del CT-103, que actúa como DNS autoritativo del laboratorio. Con `noresolv='1'` se deshabilita la consulta al `/etc/resolv.conf` local, forzando que todo el tráfico DNS pase por el servidor centralizado.

```bash
uci set dhcp.@dnsmasq[0].server='10.1.1.34'
uci set dhcp.@dnsmasq[0].noresolv='1'
uci commit dhcp
service dnsmasq restart
```

## Firewall

El firewall de OpenWRT se basa en **nftables** (en versiones modernas) con una capa de abstracción UCI que permite gestionar zonas, forwardings y reglas de forma declarativa. La filosofía es **denegar por defecto** y permitir explícitamente solo el tráfico necesario.

### Política por defecto

| Dirección | Política |
|-----------|----------|
| Input global | `REJECT` |
| Output global | `ACCEPT` |
| Forward global | `REJECT` |
| SYN flood protection | ✅ Activo |

> La protección anti-SYN flood está habilitada a nivel global para mitigar ataques de denegación de servicio básicos desde la WAN.

### Zonas

Cada VLAN se mapea a una **zona de firewall** independiente. El tráfico de entrada y salida se acepta dentro de cada zona (permitiendo que los hosts se comuniquen con el router), pero el **forward entre zonas está bloqueado** por defecto (`REJECT`).

La zona `wan` tiene habilitado **masquerading** (NAT de origen), lo que permite que los hosts internos salgan a internet con la IP pública del router.

| Zona | Red | Input | Output | Forward | Masquerade |
|------|-----|-------|--------|---------|------------|
| `wan` | `eth0` | REJECT | ACCEPT | REJECT | Sí |
| `vlan10` | `eth1.10` | ACCEPT | ACCEPT | REJECT | No |
| `vlan20` | `eth1.20` | ACCEPT | ACCEPT | REJECT | No |
| `vlan30` | `eth1.30` | ACCEPT | ACCEPT | REJECT | No |
| `vlan40` | `eth1.40` | ACCEPT | ACCEPT | REJECT | No |
| `vlan50` | `eth1.50` | ACCEPT | ACCEPT | REJECT | No |

### Forwardings (tráfico permitido entre zonas)

Los **forwardings** definen qué zonas pueden enrutar tráfico hacia otras. En este diseño, todas las VLANs tienen salida hacia `wan` (internet), pero **ninguna VLAN puede acceder directamente a otra** — el tráfico inter-VLAN está completamente segmentado.

| Origen | Destino | Descripción |
|--------|---------|-------------|
| `vlan10` | `wan` | DMZ → Internet |
| `vlan20` | `wan` | Servicios → Internet |
| `vlan30` | `wan` | SOC → Internet |
| `vlan40` | `wan` | Producción → Internet |
| `vlan50` | `wan` | ??? → Internet |

>  El tráfico entre VLANs está bloqueado (`forward=REJECT`). Ninguna VLAN puede acceder a otra directamente. Cualquier comunicación inter-VLAN que sea necesaria deberá gestionarse con reglas explícitas adicionales.

### Reglas activas

Las siguientes reglas permiten tráfico específico de control y gestión desde la WAN, necesario para el correcto funcionamiento del stack de red (DHCP, ICMP, IPv6):

| Nombre | Origen | Protocolo | Puerto | Acción |
|--------|--------|-----------|--------|--------|
| `Allow-DHCP-Renew` | wan | UDP | 68 | ACCEPT |
| `Allow-Ping` | wan | ICMP | echo-request | ACCEPT |
| `Allow-IGMP` | wan | IGMP | — | ACCEPT |
| `Allow-DHCPv6` | wan | UDP | 546 | ACCEPT |
| `Allow-MLD` | wan | ICMPv6 | 130-143 | ACCEPT |
| `Allow-ICMPv6-Input` | wan | ICMPv6 | varios | ACCEPT |
| `Allow-ICMPv6-Forward` | wan | ICMPv6 | varios | ACCEPT |

### Comandos para aplicar el firewall

Los siguientes comandos crean las zonas y los forwardings mediante UCI. El proceso debe repetirse para cada VLAN (`vlan20`, `vlan30`, `vlan40`, `vlan50`) con los valores correspondientes.

```bash
# Zonas — ejemplo para VLAN 10 (repetir para el resto)
uci add firewall zone
uci set firewall.@zone[-1].name='vlan10'
uci set firewall.@zone[-1].network='vlan10'
uci set firewall.@zone[-1].input='ACCEPT'
uci set firewall.@zone[-1].output='ACCEPT'
uci set firewall.@zone[-1].forward='REJECT'

# (repetir para vlan20, vlan30, vlan40, vlan50)

# Forwardings — ejemplo para VLAN 10 (repetir para el resto)
uci add firewall forwarding
uci set firewall.@forwarding[-1].src='vlan10'
uci set firewall.@forwarding[-1].dest='wan'

# (repetir para vlan20, vlan30, vlan40, vlan50)

uci commit firewall
service firewall restart
```

## Comandos útiles

Referencia rápida para operar y diagnosticar el VM-201 desde la shell de Proxmox o desde dentro del propio contenedor.

```bash
# Entrar al CT desde Proxmox
pct exec 100 -- sh

# Ver interfaces y direcciones IP asignadas
ip a

# Ver configuración UCI completa
uci show network
uci show dhcp
uci show firewall

# Ver leases DHCP activos (clientes conectados y sus IPs)
cat /tmp/dhcp.leases

# Reiniciar servicios individualmente
service network restart
service dnsmasq restart
service firewall restart
```
