# CT 105 — Nginx Reverse Proxy

> Este contenedor actúa como punto de entrada único para todos los servicios web del laboratorio. En lugar de exponer cada servicio directamente al exterior, el tráfico pasa por Nginx, que lo redirige al CT o VM correspondiente. Esto permite centralizar el acceso, añadir autenticación y simplificar las reglas de firewall.

## Datos del contenedor

> Contenedor no privilegiado (más seguro que el CT101), con recursos mínimos ya que Nginx es muy ligero. Al ser el punto de entrada de tráfico externo, corre en la VLAN 20 (Servicios), separada de la VLAN SOC donde están los servicios internos.

## Nginx

### Instalación

> `nginx` es el servidor web/proxy y `apache2-utils` proporciona la herramienta `htpasswd`, necesaria para gestionar los ficheros de autenticación Basic Auth de Homepage.

```bash
apt update && apt install -y nginx apache2-utils
```

### Virtual host Grafana `/etc/nginx/sites-available/grafana`

> Proxy hacia el Grafana del CT101. Los headers `proxy_set_header` transmiten la IP real del cliente a Grafana, que de otra forma solo vería la IP del proxy. El bloque de WebSockets (`Upgrade` / `Connection`) es imprescindible para que funcione Grafana Live (actualizaciones en tiempo real de los dashboards).

```nginx
server {
    listen 3000;
    server_name _;

    location / {
        proxy_pass http://10.1.1.66:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # WebSockets para Grafana Live
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

### Virtual host SOAR `/etc/nginx/sites-available/soar`

> Proxy hacia la plataforma SOAR del CT104. Configuración básica sin WebSockets ya que SOAR no los requiere. Escucha en el puerto `8080` tanto en el proxy como en el destino.

```nginx
server {
    listen 8080;
    server_name _;

    location / {
        proxy_pass http://10.1.1.37:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### Virtual host Vaultwarden `/etc/nginx/sites-available/vaultwarden`

> Proxy hacia Vaultwarden (gestor de contraseñas). El soporte WebSocket es obligatorio para Vaultwarden: lo usa para sincronizar el estado en tiempo real entre los clientes. Nótese que el puerto interno de Docker es el `8090`, pero el acceso externo se hace por el `8091` para evitar conflicto con el SOAR.

```nginx
server {
    listen 8091;
    server_name _;

    location / {
        proxy_pass http://10.1.1.80:8090;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # MUY IMPORTANTE (Vaultwarden)
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

> Puerto interno de Vaultwarden: **8090** (Docker). Puerto externo de acceso: **8091**.

### Virtual host Homepage `/etc/nginx/sites-available/homepage`

> Proxy hacia el dashboard Homepage del CT107. Es el único servicio con autenticación Basic Auth activada: Nginx solicita usuario y contraseña antes de redirigir la petición. Las credenciales se almacenan en `/etc/nginx/.htpasswd` en formato hasheado. Los WebSockets son necesarios para que Homepage actualice los widgets dinámicamente.

```nginx
server {
    listen 8888;
    server_name _;

    auth_basic "SOC Dashboard";
    auth_basic_user_file /etc/nginx/.htpasswd;

    location / {
        proxy_pass http://10.1.1.68:3001;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

> Autenticación Basic Auth via htpasswd. Fichero: `/etc/nginx/.htpasswd`.  
> Usuario: `admin`. Para regenerar: `htpasswd /etc/nginx/.htpasswd admin`

### Virtual host Web (reservado) `/etc/nginx/sites-available/web`

> Virtual host por defecto en el puerto `80` que devuelve un `503` a cualquier petición. Sirve como placeholder para un posible portal web futuro y evita que Nginx muestre su página por defecto si alguien accede al puerto estándar HTTP.

```nginx
server {
    listen 80 default_server;
    server_name _;

    location / {
        return 503 "Próximamente";
    }
}
```

### Activar y recargar

> En Nginx, los sitios disponibles se definen en `sites-available/` pero solo se activan creando un enlace simbólico en `sites-enabled/`. El comando `nginx -t` valida la sintaxis antes de recargar, evitando dejar el proxy caído por un error de configuración.

```bash
# Activar todos los vhosts
ln -s /etc/nginx/sites-available/grafana /etc/nginx/sites-enabled/
ln -s /etc/nginx/sites-available/soar /etc/nginx/sites-enabled/
ln -s /etc/nginx/sites-available/vaultwarden /etc/nginx/sites-enabled/
ln -s /etc/nginx/sites-available/homepage /etc/nginx/sites-enabled/
ln -s /etc/nginx/sites-available/web /etc/nginx/sites-enabled/

# Verificar y recargar
nginx -t && systemctl reload nginx
```

## Port forwarding en Proxmox

> Las reglas `iptables` en el host Proxmox (`honeycos`) redirigen el tráfico que llega por la interfaz externa (`vmbr0`) hacia el CT105. Sin estas reglas, el tráfico externo nunca llegaría a la red interna. Se usan tres tipos de reglas: `PREROUTING` (redirige el destino), `MASQUERADE` (necesario para que el tráfico de retorno funcione correctamente en NAT) y `FORWARD` (permite el paso del tráfico entre interfaces). Las reglas se persisten con `netfilter-persistent` para sobrevivir reinicios.

```bash
# PREROUTING — redirigir tráfico externo a CT105
iptables -t nat -A PREROUTING -i vmbr0 -p tcp --dport 3000 -j DNAT --to 10.1.1.35:3000
iptables -t nat -A PREROUTING -i vmbr0 -p tcp --dport 8080 -j DNAT --to 10.1.1.35:8080
iptables -t nat -A PREROUTING -i vmbr0 -p tcp --dport 8091 -j DNAT --to 10.1.1.35:8091
iptables -t nat -A PREROUTING -i vmbr0 -p tcp --dport 8443 -j DNAT --to 10.1.1.35:8443
iptables -t nat -A PREROUTING -i vmbr0 -p tcp --dport 8888 -j DNAT --to 10.1.1.35:8888

# MASQUERADE — necesario para retorno de tráfico NAT
iptables -t nat -A POSTROUTING -d 10.1.1.35 -p tcp --dport 8091 -j MASQUERADE
iptables -t nat -A POSTROUTING -d 10.1.1.35 -p tcp --dport 8443 -j MASQUERADE
iptables -t nat -A POSTROUTING -d 10.1.1.35 -p tcp --dport 8888 -j MASQUERADE

# FORWARD
iptables -A FORWARD -d 10.1.1.35 -p tcp --dport 8091 -j ACCEPT
iptables -A FORWARD -d 10.1.1.35 -p tcp --dport 8443 -j ACCEPT
iptables -A FORWARD -d 10.1.1.35 -p tcp --dport 8888 -j ACCEPT

# Wazuh — acceso directo sin pasar por Nginx
iptables -t nat -A PREROUTING -i vmbr0 -p tcp --dport 443 -j DNAT --to 10.1.1.67:443

# Guardar
netfilter-persistent save
```

## Comandos útiles

> Referencia rápida para operar Nginx. `systemctl reload` es preferible a `restart` porque recarga la configuración sin cortar las conexiones activas. Los logs de acceso y error son el primer lugar donde mirar si algo no funciona correctamente.

```bash
# Ver estado
systemctl status nginx

# Verificar configuración
nginx -t

# Recargar (sin cortar conexiones)
systemctl reload nginx

# Reiniciar
systemctl restart nginx

# Ver logs
tail -f /var/log/nginx/access.log
tail -f /var/log/nginx/error.log

# Gestionar htpasswd (Homepage)
htpasswd /etc/nginx/.htpasswd admin        # cambiar password
htpasswd -c /etc/nginx/.htpasswd admin     # recrear fichero
```
