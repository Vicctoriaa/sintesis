## 2. Arquitectura DNS

> Flujo de resolución DNS del laboratorio. Los contenedores de la VLAN SOC consultan a Bind9, que resuelve internamente los nombres de `soc.local` y delega al exterior (Google/Cloudflare) cualquier dominio que no conozca. Bind9 actúa como servidor maestro (authoritative) para las dos zonas internas.

```
Contenedores SOC
      │
      ▼
Bind9 (10.1.1.34)
      │
      ├── Zona interna: soc.local (master)
      ├── Zona inversa: 1.1.10.in-addr.arpa (master)
      │
      └── Forwarders externos
            ├── 8.8.8.8 (Google)
            └── 1.1.1.1 (Cloudflare)
```

## 3. Configuración principal

### `named.conf.options`

> Configuración global del comportamiento de BIND9. La recursión está habilitada pero restringida a la red interna (`10.1.1.0/24`) para evitar que el servidor sea usado como open resolver desde el exterior. Los forwarders se usan cuando Bind9 no puede resolver una consulta localmente. El bloque `statistics-channels` expone métricas internas que el `bind_exporter` de Prometheus consume.

```
options {
    directory "/var/cache/bind";
    recursion yes;
    allow-recursion { 10.1.1.0/24; localhost; };
    allow-query { 10.1.1.0/24; localhost; };
    forwarders {
        8.8.8.8;
        1.1.1.1;
    };
    dnssec-validation auto;
    listen-on { 10.1.1.34; localhost; };
};

statistics-channels {
    inet 127.0.0.1 port 8080 allow { 127.0.0.1; };
};
```

### `named.conf.local`

> Define las zonas DNS que este servidor gestiona de forma autoritativa. La zona directa `soc.local` resuelve nombres a IPs, y la zona inversa `1.1.10.in-addr.arpa` resuelve IPs a nombres (registros PTR). Ambas apuntan a sus respectivos ficheros de zona en `/etc/bind/zones/`.

```
zone "soc.local" {
    type master;
    file "/etc/bind/zones/db.soc.local";
};

zone "1.1.10.in-addr.arpa" {
    type master;
    file "/etc/bind/zones/db.10.1.1";
};
```

## 4. Zona directa — `soc.local`

> La zona directa es la que resuelve nombres a IPs (`hostname → IP`). El registro SOA (Start of Authority) define metadatos de la zona como el serial, que debe incrementarse cada vez que se modifica el fichero para que los clientes sepan que hay cambios. El TTL de 7 días indica cuánto tiempo los clientes pueden cachear las respuestas.

**Fichero:** `/etc/bind/zones/db.soc.local`
**Serial:** `2026041301`
**TTL:** 604800 (7 días)

```
$TTL    604800
@       IN      SOA     dns.soc.local. admin.soc.local. (
                        2026041301
                        604800
                        86400
                        2419200
                        604800 )
@       IN      NS      dns.soc.local.

; VLAN 10 - DMZ (10.1.1.0/27)
openwrt-fw      IN  A   10.1.1.1

; VLAN 20 - Servicios (10.1.1.32/27)
dns             IN  A   10.1.1.34
nginx           IN  A   10.1.1.35
suricata        IN  A   10.1.1.36
soar            IN  A   10.1.1.37

; VLAN 30 - SOC (10.1.1.64/27)
grafana         IN  A   10.1.1.66
prometheus      IN  A   10.1.1.66
wazuh           IN  A   10.1.1.67
homepage        IN  A   10.1.1.68
```

## 5. Zona inversa — `1.1.10.in-addr.arpa`

> La zona inversa resuelve IPs a nombres (`IP → hostname`), lo que se conoce como resolución inversa o PTR lookup. Es útil para herramientas de seguridad, logs de sistema y diagnóstico de red que muestran hostnames en lugar de IPs. El nombre de la zona (`1.1.10.in-addr.arpa`) es la notación estándar de DNS para la subred `10.1.1.x` escrita al revés.

**Fichero:** `/etc/bind/zones/db.10.1.1`
**Serial:** `2026041301`
**TTL:** 604800 (7 días)

```
$TTL 604800
@   IN  SOA     dns.soc.local. admin.soc.local. (
                2026041301  ; Serial
                604800      ; Refresh
                86400       ; Retry
                2419200     ; Expire
                604800 )    ; Negative Cache TTL
@       IN  NS      dns.soc.local.

; VLAN 10 - DMZ
1       IN  PTR     openwrt-fw.soc.local.

; VLAN 20 - Servicios
34      IN  PTR     dns.soc.local.
35      IN  PTR     nginx.soc.local.
36      IN  PTR     suricata.soc.local.
37      IN  PTR     soar.soc.local.

; VLAN 30 - SOC
66      IN  PTR     grafana.soc.local.
67      IN  PTR     wazuh.soc.local.
68      IN  PTR     homepage.soc.local.
80      IN  PTR     vaultwarden.soc.local.

; VLAN 40 - Produccion
98      IN  PTR     ldap.soc.local.

; VLAN 50 - Honeypot
130     IN  PTR     honeypot.soc.local.
```

## 6. Comandos de gestión

> Referencia rápida para las operaciones más habituales. `rndc reload` es el comando preferido para aplicar cambios en las zonas sin interrumpir el servicio. Los comandos `named-check*` permiten validar la sintaxis antes de recargar, evitando que un error en el fichero de zona tire el servicio. `dig` es la herramienta estándar para verificar que las resoluciones funcionan correctamente.

```bash
# Recargar zonas sin reiniciar
rndc reload

# Verificar sintaxis
named-checkconf
named-checkzone soc.local /etc/bind/zones/db.soc.local
named-checkzone 1.1.10.in-addr.arpa /etc/bind/zones/db.10.1.1

# Resolución directa
dig @10.1.1.34 honeypot.soc.local +short
dig @10.1.1.34 grafana.soc.local +short
dig @10.1.1.34 wazuh.soc.local +short

# Resolución inversa
dig @10.1.1.34 -x 10.1.1.130 +short
dig @10.1.1.34 -x 10.1.1.67 +short

# Comprobar forwarder externo
dig @10.1.1.34 google.com +short

# Reiniciar servicio
systemctl restart named
systemctl status named
```

## 7. Procedimiento para añadir un nuevo host

> Proceso estándar para registrar un nuevo CT o VM en el DNS interno. Es importante seguir el orden: primero editar los ficheros, luego verificar sintaxis y finalmente recargar. Saltarse la verificación puede dejar el servicio DNS caído si hay un error tipográfico en el fichero de zona.

1. Añadir registro A en `/etc/bind/zones/db.soc.local`
2. Añadir registro PTR en `/etc/bind/zones/db.10.1.1`
3. Incrementar el serial en ambos ficheros (formato `YYYYMMDDnn`)
4. Verificar sintaxis: `named-checkzone soc.local /etc/bind/zones/db.soc.local`
5. Recargar: `rndc reload`
6. Verificar: `dig @10.1.1.34 <nuevo-host>.soc.local +short`
