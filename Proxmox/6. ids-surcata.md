# Suricata — Configuración CT 106

> Este contenedor aloja Suricata, el motor de detección de intrusiones (IDS) del laboratorio. Suricata analiza el tráfico de red en tiempo real aplicando reglas para detectar amenazas, y escribe los eventos en formato JSON (`eve.json`). Ese fichero es consumido por el agente Wazuh, que lo envía al SIEM para correlación y visualización.

## Datos del contenedor

> Contenedor privilegiado con más recursos que el resto (2 cores, 1 GB RAM, 8 GB disco) porque Suricata es computacionalmente exigente: necesita inspeccionar cada paquete de red en tiempo real. El modo privilegiado es necesario para acceder directamente a las interfaces de red del host.

| Campo | Valor |
|-------|-------|
| CT ID | 106 |
| Hostname | suricata-ids |
| Imagen | debian-12-standard_12.12-1_amd64.tar.zst |

## Suricata + jq + Wazuh Integration Guide

> Guía completa para instalar Suricata, analizar logs con jq e integrarlo con Wazuh como SIEM.

---

## 1. Instalación de Suricata

### Actualizar sistema

> Paso previo obligatorio para asegurarse de que los repositorios están al día y que Suricata se instala en su versión más reciente disponible en Debian 12.

```bash
apt update && apt upgrade -y
```

### Instalar Suricata

> Instala Suricata y sus dependencias desde los repositorios oficiales de Debian. La versión incluida en Debian 12 es suficiente para un entorno de laboratorio.

```bash
apt install suricata -y
```

### Verificar instalación

> Confirma que Suricata se ha instalado correctamente y muestra la versión activa.

```bash
suricata -V
```

### Configurar interfaz de red

> Suricata necesita saber en qué interfaz de red escuchar el tráfico. Por defecto usa `eth0`, pero en Proxmox LXC el nombre puede variar. Es fundamental especificar la interfaz correcta o Suricata no capturará ningún paquete.

Ver interfaces disponibles:

```bash
ip a
```

Editar configuración:

```bash
nano /etc/suricata/suricata.yaml
```

Buscar sección:

```yaml
af-packet:
  - interface: eth0
```

>  Cambia `eth0` por tu interfaz real (ej: `ens33`, `ens18`, etc.)

###  Iniciar Suricata

> Habilita Suricata para que arranque automáticamente con el sistema y lo inicia inmediatamente. A partir de aquí comenzará a inspeccionar el tráfico y a escribir eventos en `eve.json`.

```bash
systemctl enable suricata
systemctl start suricata
```

### 📊 Ver logs en tiempo real

> `eve.json` es el fichero de salida principal de Suricata en formato JSON. Contiene todos los eventos: alertas, tráfico HTTP, DNS, TLS, ficheros, etc. Este comando permite verlos en tiempo real a medida que se generan.

```bash
tail -f /var/log/suricata/eve.json
```

## 2. Instalación de jq (análisis de logs)

### Instalar jq

> `jq` es un procesador de JSON en línea de comandos. Como `eve.json` es un fichero JSON con una entrada por línea, `jq` permite filtrar, buscar y formatear los eventos de Suricata de forma legible sin necesidad de herramientas externas.

```bash
apt install jq -y
```

### ✔️ Verificar instalación

```bash
jq --version
```

###  Uso básico con Suricata

> Ejemplos prácticos para filtrar eventos específicos del fichero de logs. El campo `event_type` en cada entrada JSON indica el tipo de evento que Suricata ha registrado.

Ver alertas:

```bash
jq 'select(.event_type=="alert")' /var/log/suricata/eve.json
```

Ver tráfico HTTP:

```bash
jq 'select(.event_type=="http")' /var/log/suricata/eve.json
```

Tiempo real (alertas en vivo):

```bash
tail -f /var/log/suricata/eve.json | jq 'select(.event_type=="alert")'
```

> Este último comando combina `tail -f` (streaming del fichero) con `jq` para mostrar únicamente las alertas de seguridad en tiempo real, filtrando el resto de eventos. Muy útil para monitorización activa.

---

## 3. Integración con Wazuh

### Objetivo

> El flujo de integración es el siguiente: Suricata detecta eventos en la red y los escribe en `eve.json`. El agente Wazuh, instalado en este mismo CT, lee ese fichero de forma continua y envía los eventos al manager Wazuh (VM202), que los indexa y los muestra en el dashboard SIEM. Esto permite correlacionar las alertas IDS con el resto de eventos del laboratorio.

```
Suricata genera logs en eve.json
       ↓
Wazuh Agent los lee
       ↓
Wazuh Manager los muestra en dashboard SIEM
```

### Configurar Wazuh Agent

> Se añade un bloque `<localfile>` al fichero de configuración del agente Wazuh para indicarle que debe leer y enviar el contenido de `eve.json`. El formato `json` le dice al agente cómo parsear cada línea del fichero antes de enviarlo al manager.

Editar configuración:

```bash
nano /var/ossec/etc/ossec.conf
```

Añadir:

```xml
<localfile>
  <log_format>json</log_format>
  <location>/var/log/suricata/eve.json</location>
</localfile>
```

### Reiniciar agente

> Necesario para que el agente Wazuh aplique el nuevo bloque de configuración y comience a monitorizar `eve.json`.

```bash
systemctl restart wazuh-agent
```
